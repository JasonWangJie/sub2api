# 服务端个人图库与统一对象引用

## 浏览器本地个人图库（用户侧）

工作台侧栏与 `/image-library` 的「个人图库」只展示本机 IndexedDB：

- 仅实时生图写入；异步任务成功后不进入个人图库。
- 保留近 **180 天** 或最多 **500** 条（先删过期，再删最旧）。
- 换设备 / 清除站点数据会丢失；投稿审核元数据在服务端，图片字节在审核通过后由用户 sync 上传。

## 为什么需要统一对象表

异步任务结果、投稿 sync、手工导入和公开投稿可能指向对象存储中的图片。如果各模块独立保存和删除对象，会造成重复字节、预签名 URL 过期、误删仍在使用的图片或容量统计失真。

迁移 `186_ZJ_image_library_and_plaza_moderation.sql` 引入 `image_storage_objects` 作为稳定对象身份。业务表只引用它；实际查看时再按权限动态生成公开 CDN URL、短期预签名 URL，或对本机 durable 对象直接流式返回。

```text
image_storage_objects
  <- async_image_results.storage_object_id          （异步 ephemeral 桶）
  <- image_library_items.storage_object_id         （投稿 sync / import → durable）
       <- image_plaza_publications.library_item_id

image_plaza_submission_requests  （迁移 188；审核前可无 storage_object_id）
  -> sync 成功后写入 image_storage_objects + image_library_items + published publication
     （写入 image_durable_storage，不写 image_storage）
```

`async_image_results` 原有 provider/bucket/object_key 等列保留一个兼容周期，并在迁移时回填 `storage_object_id`。不要在尚未完成兼容迁移和回归前删除旧列。

工作台不再自动 `from-task` 把异步结果写入服务端私有图库；服务端 `image_library_items` 主要作为投稿/广场持久化载体。

## `186` 新增数据结构

| 表/字段 | 用途 |
|---|---|
| `image_storage_objects` | provider、bucket、object key、实际 MIME、字节、SHA-256、宽高和删除状态 |
| `async_image_results.storage_object_id` | 将持久异步结果连接到统一对象身份 |
| `image_library_items` | 用户私有图库、来源、Key/分组、平台、模式、任务、模型、规格、提示词和保留期 |
| `image_plaza_publications` | 显式投稿及审核/下架/撤回状态（图库资产路径） |
| `image_plaza_reports` | 登录用户对已公开投稿的举报 |
| `image_library_events` | 图库和投稿状态事件、操作者和审计载荷 |
| `image_library_outbox` | 删除等外部副作用的可靠执行记录 |
| `image_library_cleanup_jobs` | 管理员和自动过期清理任务、租约、进度和错误 |
| `image_library_migration_state` | 旧广场迁移进度、租约、成功/隔离计数和错误 |

## `188` 延期投稿

| 表/字段 | 用途 |
|---|---|
| `image_plaza_submission_requests` | 本机持图延期投稿：checksum、尺寸、公开元数据、状态机；审核前不占 OSS |

状态：`pending_review` → `approved_pending_sync` →（用户 sync）`synced`；也可 `rejected` / `withdrawn`。

对外标识使用不透明的 `img_*` 和 `imgpub_*`，不能暴露数据库顺序 ID。对象 key 由服务端按 UTC `YYYY/MM/DD` 分区生成。

## 图库来源与生成模式

`image_library_items.generation_mode`：

- `realtime`：工作台实时生成后导入。
- `async`：从已成功且已结算的持久异步任务归档。
- `import`：手工导入或旧广场迁移。

`source_type`：

- `realtime_import`
- `async_task`
- `manual_import`
- `legacy_plaza`

每条图库记录还保留 API Key、分组、平台、模型、请求/实际规格、比例、质量、私有标题和私有提示词。普通公开响应不得返回内部 user ID、API Key、分组、对象 key 或未选择共享的提示词。

## 个人图库 API

所有接口要求站内用户认证；越权统一表现为 `404`。

| 方法与路径 | 行为 |
|---|---|
| `GET /api/v1/user/image-library` | 稳定游标分页；支持 `visibility/source_type/platform/status/q` |
| `POST /api/v1/user/image-library/import` | multipart `file` 导入；支持 `Idempotency-Key` |
| `POST /api/v1/user/image-library/import-url` | 下载上游 HTTPS URL 并归档；支持 `Idempotency-Key` |
| `POST /api/v1/user/image-library/from-task` | 归档本人的成功且已结算异步结果 |
| `GET /api/v1/user/image-library/:asset_id` | 获取本人图库详情 |
| `PATCH /api/v1/user/image-library/:asset_id` | 编辑私有标题/提示词 |
| `DELETE /api/v1/user/image-library/:asset_id` | 软删除并通过 Outbox 触发引用清理 |
| `GET /api/v1/user/image-library/:asset_id/view` | 默认 `307` 跳转；JSON Accept 返回 URL 和到期时间 |
| `POST /api/v1/user/image-library/:asset_id/publications` | 显式投稿，选择公开标题和是否共享提示词 |
| `DELETE /api/v1/user/image-library/:asset_id/publication` | 撤回待审或已公开投稿 |
| `GET /api/v1/user/image-library/submission-requests` | 列出本人延期投稿队列 |
| `POST /api/v1/user/image-library/submission-requests` | 本机持图投稿（仅元数据，不上传 OSS） |
| `POST /api/v1/user/image-library/submission-requests/:request_id/sync` | 审核通过后上传并发布到广场 |
| `DELETE /api/v1/user/image-library/submission-requests/:request_id` | 撤回待审或待同步请求 |

列表使用 `(created_at, id)` 稳定游标，不用页码加 offset。`from-task` 只允许任务所有者，并要求任务 `succeeded`、账务 `succeeded/not_billable` 和对应结果存在。

工作台实时结果优先走 `submission-requests`；已归档到图库的资产继续走 `publications`。

## 导入与幂等

### 二进制导入

multipart 请求在读取前套用最大 body 限制。服务端不信任文件名、扩展名或客户端 MIME；只保存从图片字节中得出的真实 MIME、尺寸、SHA-256 和字节数。

### URL 导入

只能使用受安全下载策略允许的 URL。初始 DNS 和每次重定向都拒绝内网、回环、链路本地和保留地址，同时限制协议、重定向、超时、字节、像素和 MIME，防止 SSRF 与 DNS rebinding。

### 幂等作用域

导入幂等以用户和 `Idempotency-Key` 为作用域：同一 key、同一请求哈希返回原资产；同一 key 配不同元数据返回冲突。异步归档另由 `(user_id, task_id, image_index)` 保证唯一。

## 默认配额和保留期

| 配置字段 | 默认值 |
|---|---:|
| `image_library.retention_days` | 90 天 |
| `image_library.max_items_per_user` | 1000 |
| `image_library.max_bytes_per_user` | 5 GiB |
| `image_library.max_image_bytes` | 20 MiB |
| `image_library.max_image_pixels` | 40,000,000 |
| `image_library.signed_url_expiry_seconds` | 3600 秒 |
| `image_library.import_per_minute` | 20 |
| `image_library.publish_per_minute` | 10 |

配额限制归档/导入，不得把已经成功的模型调用改成失败，也不得重新生成。批准公开时，公开记录的到期时间至少覆盖批准后的默认保留窗口；具体实现变动要通过保留期测试确认。

## 对象查看

数据库永远不保存会过期的预签名 URL。查看流程：

```text
登录态/公开状态鉴权
  -> 读取稳定 ObjectRef
  -> 当前 OSS 配置解析对象
  -> 生成公开 CDN URL 或短期签名 URL
  -> 307 Redirect
```

前端需要直接取得 URL 时发送 `Accept: application/json`，服务端返回：

```json
{
  "url": "https://...",
  "expires_at": "2026-07-21T12:00:00Z"
}
```

页面应使用稳定站内查看地址，不把一次签名结果长期持久化。

## 引用安全删除

删除 OSS 前必须确认不存在以下任何活动引用：

- 尚未删除的 `image_library_items`。
- `pending_review`、`published` 或 `admin_hidden` 且未过期的投稿。
- 任意 `async_image_results`，包括迁移兼容期内仅凭 provider/bucket/object_key 匹配的记录。

删除采用 `active -> deleting -> deleted` 状态和 claim。OSS 删除成功后数据库必须同步标记对象删除；数据库更新失败时，stale deletion 恢复逻辑要能继续收敛。异步结果保留清理也必须更新统一对象状态，避免容量统计残留 active 对象。

## 管理 API

| 方法与路径 | 行为 |
|---|---|
| `GET /api/v1/admin/image-library` | 全站图库查询 |
| `GET /api/v1/admin/image-library/stats` | 条目、对象、字节、待审、公开、举报统计 |
| `GET /api/v1/admin/image-library/:asset_id/view` | 管理员动态查看 |
| `GET /api/v1/admin/image-library/cleanup-jobs` | 清理任务列表 |
| `POST /api/v1/admin/image-library/cleanup-jobs/preview` | 按 scope/filter 预览条目数和字节 |
| `POST /api/v1/admin/image-library/cleanup-jobs` | 创建 `expired/deleted/user` 范围的清理任务 |
| `GET /api/v1/admin/image-library/migration` | 查看旧广场迁移进度和隔离数 |

任何清理 UI 都应先预览再创建任务。执行结果以数据库 job 为准，不以浏览器请求是否保持连接为准。
