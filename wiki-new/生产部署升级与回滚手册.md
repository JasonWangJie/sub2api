# 生产部署、升级与回滚操作手册

本文说明如何把 `JasonWangJie/sub2api` Fork 的当前 `main` 部署到 Linux 服务器。推荐架构是 Docker Compose + PostgreSQL + Redis + Caddy/Nginx。生产镜像内已嵌入前端，不需要单独部署 Vue 静态站点。

## 1. 关键结论

1. 必须从 Fork 当前 `main` 构建镜像，或使用你自己发布并固定 digest 的 Fork 镜像。
2. 不要直接使用 Compose 默认的 `weishaw/sub2api:latest`，它是原作者镜像，不能保证包含本 Fork 的图片工作流和迁移 `185` 到 `188`。
3. 不要使用浮动 `latest` 作为生产回滚依据。每次镜像标签至少包含 Git SHA。
4. PostgreSQL 是持久异步任务、账务、图库和审核状态的事实源；Redis 不是数据库替代品。
5. 迁移自动执行且是 forward-only。升级前必须备份 PostgreSQL，不能把“切回旧镜像”等同于完整回滚。
6. 不得提交 `.env`、`config.yaml`、OSS 密钥、JWT 密钥或 TOTP 加密密钥。

## 2. 推荐拓扑

```text
Internet
  -> DNS / 可选 CDN-WAF
  -> Caddy 或 Nginx :443
  -> Sub2API :8080（只监听 127.0.0.1）
       -> PostgreSQL 18（仅 Compose 内网）
       -> Redis 8（仅 Compose 内网）
       -> Gemini / OpenAI / Grok 上游
       -> 七牛 / 阿里 / 腾讯 / S3 兼容对象存储
```

单机推荐使用 `deploy/docker-compose.local.yml` 的宿主机目录持久化模式，便于备份和迁移。数据库和 Redis 不应暴露公网端口。

## 3. 服务器要求

- 64 位 Linux，推荐 Ubuntu 22.04/24.04 或 Debian 12。
- Docker Engine 24+ 和 Docker Compose v2。
- Git、curl、openssl、tar、gzip。
- 至少 2 CPU、4 GiB 内存；启用多 Worker、4K 图片或高并发时应提高内存和数据库磁盘容量。
- 独立持久磁盘；需监控 PostgreSQL 数据、WAL、异步 staging、日志和 OSS 容量。
- 已解析到服务器的域名，开放 80/443；8080 只绑定回环地址。

检查：

```bash
docker version
docker compose version
git --version
df -h
free -h
```

## 4. 首次部署

### 4.1 获取 Fork 主分支

```bash
sudo mkdir -p /opt/sub2api
sudo chown "$USER":"$USER" /opt/sub2api
git clone https://github.com/JasonWangJie/sub2api.git /opt/sub2api/source
cd /opt/sub2api/source
git switch main
git pull --ff-only origin main
git status --short --branch
git rev-parse HEAD
cat backend/cmd/server/VERSION
```

记录完整 SHA。部署操作只针对干净的 `main` 工作树。

### 4.2 构建带 SHA 的 Fork 镜像

```bash
cd /opt/sub2api/source
FULL_SHA="$(git rev-parse HEAD)"
SHORT_SHA="$(git rev-parse --short=12 HEAD)"
VERSION="$(cat backend/cmd/server/VERSION)"
BUILD_DATE="$(date -u +%Y-%m-%dT%H:%M:%SZ)"

docker build --pull \
  --build-arg VERSION="$VERSION" \
  --build-arg COMMIT="$FULL_SHA" \
  --build-arg DATE="$BUILD_DATE" \
  -t "sub2api-fork:$SHORT_SHA" \
  .

docker image inspect "sub2api-fork:$SHORT_SHA" --format '{{.Id}}'
```

镜像构建过程会先构建 Vue 前端，再用 `-tags embed` 编译 Go 后端，最后生成一个运行镜像。

### 4.3 准备 Compose override

进入部署目录：

```bash
cd /opt/sub2api/source/deploy
cp .env.example .env
chmod 600 .env
```

创建 `deploy/docker-compose.override.yml`。该路径已被仓库忽略，不会污染生产服务器的 Git 工作树：

```yaml
services:
  sub2api:
    image: sub2api-fork:${SUB2API_FORK_TAG}
    pull_policy: never
```

这样保留原 Compose 的数据库、Redis、健康检查和目录挂载，只替换应用镜像。不要把本地 Fork 镜像伪装成 `weishaw/sub2api:latest`。

### 4.4 配置 `.env`

至少设置：

```dotenv
SUB2API_FORK_TAG=替换为上一步SHORT_SHA
BIND_HOST=127.0.0.1
SERVER_PORT=8080
SERVER_MODE=release
RUN_MODE=standard
TZ=Asia/Shanghai

POSTGRES_USER=sub2api
POSTGRES_PASSWORD=强随机数据库密码
POSTGRES_DB=sub2api
DATABASE_MAX_OPEN_CONNS=50
DATABASE_MAX_IDLE_CONNS=10
DATABASE_CONN_MAX_LIFETIME_MINUTES=30
DATABASE_CONN_MAX_IDLE_TIME_MINUTES=5

REDIS_PASSWORD=强随机Redis密码
REDIS_DB=0
REDIS_POOL_SIZE=256
REDIS_MIN_IDLE_CONNS=10

ADMIN_EMAIL=你的管理员邮箱
ADMIN_PASSWORD=强随机管理员初始密码
JWT_SECRET=至少64个十六进制字符
TOTP_ENCRYPTION_KEY=另一个至少64个十六进制字符
```

生成密钥：

```bash
openssl rand -hex 32
openssl rand -hex 32
openssl rand -base64 36
openssl rand -base64 36
```

前两项分别用于 JWT 和 TOTP，后两项可用于数据库、Redis 或管理员密码。不要复用同一密钥。

小规格服务器不要照抄 `.env.example` 中面向高并发的超大连接池。所有应用实例的 `DATABASE_MAX_OPEN_CONNS` 总和必须低于 PostgreSQL `max_connections` 并留出管理连接。

### 4.5 创建数据目录并校验配置

```bash
mkdir -p data postgres_data redis_data backups

docker compose \
  -f docker-compose.local.yml \
  -f docker-compose.override.yml \
  config > /tmp/sub2api-compose.rendered.yml

grep -n 'image: sub2api-fork' /tmp/sub2api-compose.rendered.yml
```

必须看到 `sub2api-fork:<SHA>`，不能看到应用最终仍使用 `weishaw/sub2api:latest`。

### 4.6 首次启动

```bash
docker compose \
  -f docker-compose.local.yml \
  -f docker-compose.override.yml \
  up -d

docker compose \
  -f docker-compose.local.yml \
  -f docker-compose.override.yml \
  ps

docker compose \
  -f docker-compose.local.yml \
  -f docker-compose.override.yml \
  logs -f sub2api
```

首次启动会自动创建配置、管理员和 `schema_migrations`，并按文件名执行全部迁移。等待日志稳定后验证：

```bash
curl -fsS http://127.0.0.1:8080/health
docker exec sub2api /app/sub2api -version
```

如果 `.env` 没有填写 `ADMIN_PASSWORD`，初始密码会出现在首次启动日志中。生产环境推荐显式设置，登录后立即轮换并启用双因素认证。

## 5. HTTPS 反向代理

### 5.1 Caddy

仓库提供 `deploy/Caddyfile`。复制前必须把 `api.sub2api.com` 改为真实域名：

```bash
sudo cp /opt/sub2api/source/deploy/Caddyfile /etc/caddy/Caddyfile
sudo editor /etc/caddy/Caddyfile
sudo caddy validate --config /etc/caddy/Caddyfile
sudo systemctl reload caddy
```

确认：

- DNS A/AAAA 已指向服务器。
- 防火墙开放 80/443。
- `.env` 使用 `BIND_HOST=127.0.0.1`。
- 反向代理允许长连接和流式响应。
- 请求体上限覆盖允许的参考图上传大小；仓库 Caddy 基线为 256 MiB。

验证：

```bash
curl -fsS https://你的域名/health
```

若 Caddy 前还有 CDN/WAF，必须按 `deploy/EDGE_SECURITY.md` 配置可信代理 CIDR、真实客户端 IP 和入口限制，不能无条件信任客户端传入的 `X-Forwarded-For`。

### 5.2 Nginx 要点

使用 Nginx 时至少配置：

- `proxy_http_version 1.1`。
- 关闭对 SSE/流式响应的代理缓冲。
- 合理提高 `proxy_read_timeout`。
- 限制请求体，但上限不能低于业务允许的图片上传大小。
- 只向应用传递由可信代理生成的真实 IP 头。
- TLS 1.2/1.3 和自动证书续期。

## 6. 首次登录后的必要配置

### 6.1 对象存储

进入管理员后台的备份/图片存储设置：

1. 选择全站当前供应商：七牛、阿里、腾讯或 `custom_s3`。
2. 填写 region、endpoint、bucket、access key、secret key、prefix 和寻址方式。
3. 私有桶可留空 `public_base_url`，由服务动态签发短期 URL；公开桶或 CDN 可填写固定根地址。
4. 执行测试连接。上传、读取或 HEAD、删除任一步失败都不能视为可用。
5. 配置输入、任务、结果和图库保留期；确认 OSS 生命周期规则不会早于应用保留策略删除活动对象。

保存 OSS secret 前必须保证 `TOTP_ENCRYPTION_KEY` 固定。更换该密钥会导致数据库中已有加密凭据无法解密。

### 6.2 持久异步生图

- `public_base_url` 填外部 HTTPS API 根地址，例如 `https://api.example.com`，不要带结尾 `/`。
- 单实例先使用较小 `worker_concurrency`，观察上游限制、数据库和 OSS 后再提高。
- 多实例总并发是各实例 Worker 数之和。
- 配置执行超时、存储重试、计费重试、签名时长和保留期。
- 只有 Gemini/OpenAI 且普通生图已开启的分组才能开启“异步生图”。

### 6.3 账号、分组与计费

1. 添加 Gemini/OpenAI/Grok 上游账号。
2. 校验模型映射和账号可用性。
3. 创建分组并配置图片规格价格、倍率、并发和 API Key 权限。
4. 分别验证实时与异步模式；工作台模式只能由所选 Key 当前分组决定。
5. 用小额测试逐笔核对余额、订阅、倍率、图片数量和实际尺寸。

## 7. 部署后验收

### 7.1 基础检查

```bash
cd /opt/sub2api/source/deploy

docker compose -f docker-compose.local.yml -f docker-compose.override.yml ps
docker compose -f docker-compose.local.yml -f docker-compose.override.yml logs --tail=200 sub2api
curl -fsS http://127.0.0.1:8080/health
curl -fsS https://你的域名/health
docker exec sub2api /app/sub2api -version
```

### 7.2 迁移检查

```bash
docker exec sub2api-postgres \
  psql -U sub2api -d sub2api -c \
  "SELECT filename, applied_at FROM schema_migrations WHERE filename IN ('182_ZJ_add_image_plaza.sql','185_ZJ_async_image_tasks.sql','186_ZJ_image_library_and_plaza_moderation.sql','187_ZJ_async_image_upload_reservations.sql','188_ZJ_plaza_submission_deferred_upload.sql','189_ZJ_async_image_result_upload_intents.sql','182_add_image_plaza.sql','185_async_image_tasks.sql','186_image_library_and_plaza_moderation.sql','187_async_image_upload_reservations.sql','188_plaza_submission_deferred_upload.sql','189_async_image_result_upload_intents.sql') ORDER BY filename;"
```

六个 `_ZJ` 文件名记录都应存在。由旧版本升级的数据库还会保留对应无标记旧名记录作为审计历史；迁移器在 checksum 完全一致时自动登记 `_ZJ` 别名。不要手工伪造 `schema_migrations` 记录，也不要修改已执行迁移的内容。

### 7.3 业务冒烟

- 管理员登录、退出、刷新令牌。
- 添加一个测试账号、分组、API Key。
- 调用一个原同步文本接口，确认旧路径未受影响。
- 用异步开关关闭的 Key 测试实时生图。
- 用异步开关开启的 Key 提交持久异步任务并轮询到终态。
- 检查 OSS 对象、任务记录、实际费用和用量日志。
- 验证个人图库、显式投稿、管理员审核、广场查看和撤回。
- 使用另一个 Key 验证不能读取第一个 Key 的公共异步任务。

## 8. 备份

### 8.1 每次升级前必须备份

```bash
cd /opt/sub2api/source/deploy
STAMP="$(date +%Y%m%d-%H%M%S)"
mkdir -p "backups/$STAMP"

docker exec sub2api-postgres \
  pg_dump -U sub2api -d sub2api -Fc \
  > "backups/$STAMP/sub2api.dump"

tar -czf "backups/$STAMP/app-data.tar.gz" data
cp .env "backups/$STAMP/env.backup"
cp docker-compose.override.yml "backups/$STAMP/docker-compose.override.yml"
git -C /opt/sub2api/source rev-parse HEAD > "backups/$STAMP/git-sha.txt"
docker inspect sub2api --format '{{.Config.Image}} {{.Image}}' \
  > "backups/$STAMP/running-image.txt"

chmod -R go-rwx "backups/$STAMP"
```

不要在 PostgreSQL 正在运行时直接打包 `postgres_data` 作为唯一数据库备份。优先使用 `pg_dump` 或受验证的物理备份方案。

备份目录包含数据库和生产密钥，必须再做加密、离机或异地备份，并限制只有运维账号可读。把备份留在同一块服务器磁盘上不能防止磁盘损坏或主机丢失。

Redis 中包含队列和缓存状态，可额外执行：

```bash
docker exec sub2api-redis redis-cli BGSAVE
```

但持久异步任务事实仍以 PostgreSQL 为准。Redis 备份不能替代 PostgreSQL 备份。

OSS 不在本机目录中。生产应启用桶版本控制或供应商备份，并记录生命周期规则、跨区域复制和恢复方法。

### 8.2 定期验证备份

备份只有在隔离环境成功恢复过才算有效。至少定期验证：

- `pg_restore --list` 可以读取 dump。
- 数据库恢复后 `schema_migrations` 完整。
- `data/config.yaml`、`.installed` 和固定密钥可恢复。
- 恢复环境能解析历史 OSS 对象。
- 管理员可以登录，旧 API Key、账务和任务可查询。

## 9. 升级流程

```bash
cd /opt/sub2api/source
git fetch origin
git switch main
git pull --ff-only origin main
git status --short --branch

NEW_FULL_SHA="$(git rev-parse HEAD)"
NEW_SHORT_SHA="$(git rev-parse --short=12 HEAD)"
NEW_VERSION="$(cat backend/cmd/server/VERSION)"
```

先阅读新迁移和交接文档，按第 8 节备份。然后构建新镜像：

```bash
docker build --pull \
  --build-arg VERSION="$NEW_VERSION" \
  --build-arg COMMIT="$NEW_FULL_SHA" \
  --build-arg DATE="$(date -u +%Y-%m-%dT%H:%M:%SZ)" \
  -t "sub2api-fork:$NEW_SHORT_SHA" \
  /opt/sub2api/source
```

编辑 `deploy/.env` 的 `SUB2API_FORK_TAG`，再执行：

```bash
cd /opt/sub2api/source/deploy
docker compose -f docker-compose.local.yml -f docker-compose.override.yml config > /tmp/sub2api-compose.rendered.yml
grep -n "image: sub2api-fork:$NEW_SHORT_SHA" /tmp/sub2api-compose.rendered.yml

docker compose -f docker-compose.local.yml -f docker-compose.override.yml up -d --no-deps sub2api
docker compose -f docker-compose.local.yml -f docker-compose.override.yml logs -f --tail=200 sub2api
```

启动时会自动执行尚未应用的迁移。完成第 7 节验收后，再清理很旧且确认不会回滚到的镜像。

不要在升级时执行：

```text
docker compose down -v
git reset --hard
docker system prune -a --volumes
```

这些操作可能删除持久数据、覆盖本地配置或移除回滚镜像。

## 10. 回滚

### 10.1 仅回滚应用镜像

只有在确认新版本没有执行与旧版本不兼容的迁移、没有写入旧版本无法理解的数据时，才可只切回旧镜像：

1. 把 `.env` 中 `SUB2API_FORK_TAG` 改回旧 SHA。
2. 执行 `docker compose -f docker-compose.local.yml -f docker-compose.override.yml config`，确认旧镜像。
3. 执行 `docker compose -f docker-compose.local.yml -f docker-compose.override.yml up -d --no-deps sub2api`。
4. 验证健康、登录、旧同步接口和任务查询。

### 10.2 数据库恢复回滚

迁移是 forward-only。发生不兼容迁移或数据写坏时，安全回滚通常需要：

1. 停止 Sub2API 应用，保留 PostgreSQL 和 Redis。
2. 额外保存故障现场数据库和日志，不能直接覆盖唯一证据。
3. 在维护窗口将数据库恢复到升级前 dump。
4. 恢复匹配的 `data/`、固定密钥和旧镜像标签。
5. 核对 OSS 中升级期间新增对象，避免数据库恢复后形成孤立对象或误删仍有引用的对象。
6. 启动旧镜像并完成完整验收。

示例恢复命令具有破坏性，只能对确认的目标数据库执行：

```bash
docker compose -f docker-compose.local.yml -f docker-compose.override.yml stop sub2api

cat backups/时间戳/sub2api.dump | docker exec -i sub2api-postgres \
  pg_restore -U sub2api -d sub2api --clean --if-exists --no-owner
```

如果存在活动连接、所有权或扩展错误，应由熟悉 PostgreSQL 的管理员在隔离恢复演练后制定正式命令，不要反复盲目执行 `--clean`。

`execution_unknown` 任务不能因回滚自动重新调用上游。需要再次生成时必须创建新任务，并明确可能产生第二次上游费用。

## 11. 使用外部 PostgreSQL/Redis

已有托管数据库时使用 `deploy/docker-compose.standalone.yml`，仍需通过 `docker-compose.override.yml` 覆盖应用镜像：

```bash
docker compose \
  -f docker-compose.standalone.yml \
  -f docker-compose.override.yml \
  config
```

在 `.env` 设置：

```dotenv
DATABASE_HOST=数据库地址
DATABASE_PORT=5432
DATABASE_USER=sub2api
DATABASE_PASSWORD=数据库密码
DATABASE_DBNAME=sub2api
DATABASE_SSLMODE=verify-full
REDIS_HOST=Redis地址
REDIS_PORT=6379
REDIS_USERNAME=
REDIS_PASSWORD=Redis密码
REDIS_ENABLE_TLS=true
```

外部数据库部署要求：

- PostgreSQL 用户有执行迁移所需的 DDL 权限。
- TLS CA 和主机名校验正确，生产不要无理由使用 `sslmode=disable`。
- 网络 ACL 只允许应用服务器访问。
- 数据库备份、PITR、Redis 持久化和故障切换已由供应商或运维验证。
- 多实例连接池总量、异步 Worker 总量和迁移并发均经过容量规划。

## 12. 日常运维命令

```bash
cd /opt/sub2api/source/deploy

# 状态
docker compose -f docker-compose.local.yml -f docker-compose.override.yml ps

# 应用日志
docker compose -f docker-compose.local.yml -f docker-compose.override.yml logs -f --tail=200 sub2api

# 重启应用，不重启数据库
docker compose -f docker-compose.local.yml -f docker-compose.override.yml restart sub2api

# 资源使用
docker stats sub2api sub2api-postgres sub2api-redis

# 健康检查
curl -fsS http://127.0.0.1:8080/health

# 当前运行版本
docker exec sub2api /app/sub2api -version
```

## 14. 常见故障

### 14.1 页面正常但没有 Fork 新功能

运行：

```bash
docker inspect sub2api --format '{{.Config.Image}}'
docker exec sub2api /app/sub2api -version
```

如果镜像是 `weishaw/sub2api:latest`，说明 Compose override 没有生效。重新执行 `docker compose -f docker-compose.local.yml -f docker-compose.override.yml config` 核对。

### 14.2 重启后登录全部失效或 OSS 凭据无法解密

检查 `JWT_SECRET` 和 `TOTP_ENCRYPTION_KEY` 是否保持首次部署值。不要重新生成后直接覆盖生产值。先恢复原密钥备份。

### 14.3 修改 `.env` 后数据库地址没有变化

首次自动安装后，`deploy/data/config.yaml` 已持久化主要配置，且配置文件优先级高于环境变量。先备份，检查后台设置和该文件，再决定修改路径。

### 14.4 异步任务一直失败

按顺序检查：

1. Key 当前分组是否为 Gemini/OpenAI，且普通生图和异步生图都开启。
2. PostgreSQL、Redis、Worker 和租约日志。
3. OSS 测试连接及 bucket 权限。
4. 上游账号、模型映射、并发和内容审核。
5. `billing_failed` 只重试账务，`storage_failed` 只重试后处理，不能重新调用上游。
6. `execution_unknown` 禁止自动重调。

### 14.5 迁移 checksum mismatch

说明数据库记录的迁移内容与当前镜像中的文件不一致。不要更新 checksum 绕过。找回首次执行时的原文件，确认 Git 历史，并把新变更写入新的迁移文件。
