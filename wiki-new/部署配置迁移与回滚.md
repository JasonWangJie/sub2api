# 部署、配置、迁移与回滚

本页记录图片工作流的配置边界和迁移语义。需要可直接执行的完整运维步骤时，先看 [本地开发运行手册.md](本地开发运行手册.md) 和 [生产部署升级与回滚手册.md](生产部署升级与回滚手册.md)。

## 上线前置条件

- PostgreSQL 15+ 和 Redis 7+ 可用，且已经完成可恢复备份。
- 应用加密密钥在换机、扩容和回滚后保持一致，否则排队异步任务的加密请求无法解密。
- 至少一个 OpenAI/Gemini/Grok 图片分组具有可用账号、普通图片权限和正确的现有图片计费配置。
- 七牛、阿里、腾讯或 `custom_s3` 的桶、region、endpoint、凭证、寻址方式和网络权限完整。
- `async_image.public_base_url`、站点外部地址、HTTPS 和反向代理配置已经确认。
- 已决定旧 `image_plaza_items` 被立即隐藏后如何安排审核，不把升级行为误判为数据丢失。

## 数据库迁移顺序

### `185_ZJ_async_image_tasks.sql`

建立：

- Group 的 `allow_async_image_generation`。
- 持久异步任务、结果、事件、Outbox、staging、参考图及任务输入引用。

它不删除旧 Redis 24 小时任务，也不改变旧同步/旧异步路由。

### `186_ZJ_image_library_and_plaza_moderation.sql`

建立：

- `image_storage_objects`
- `image_library_items`
- `image_plaza_publications`
- `image_plaza_reports`
- `image_library_events`
- `image_library_outbox`
- `image_library_cleanup_jobs`
- `image_library_migration_state`
- `async_image_results.storage_object_id`

迁移会从现有 `async_image_results` 回填统一对象，再把所有旧 `image_plaza_items.visibility='public'` 立即改为 `private`。这是安全设计：旧内容没有通过新校验和审核，升级后不能继续公开。

### `187_ZJ_async_image_upload_reservations.sql`

建立：

- `async_image_upload_attempts`：body 前 rolling-rate admission。
- `async_image_upload_reservations`：幂等、活跃字节、结果墓碑和 object intent。
- `async_image_input_url_aliases`：原 URL/重签 URL hash 到同一输入对象的所有权关联。

迁移不搬动现有 `async_image_input_objects` 字节，也不改变 SC 上传成功响应体；新服务启动后，SC 上传依赖 PostgreSQL admission，数据库不可用时返回 `503`。

### `188_ZJ_plaza_submission_deferred_upload.sql`

建立：

- `image_plaza_submission_requests`：本机持图延期投稿队列。
- 状态：`pending_review` / `approved_pending_sync` / `rejected` / `withdrawn` / `synced`。
- 创建阶段不写入 `storage_object_id`；同步成功后才关联对象与广场投稿。

目的：审核前不占用 OSS，避免恶意投稿灌满对象存储。启动后若缺表，相关投稿/同步接口会失败，需确认迁移已应用。

正常启动流程会按版本执行迁移。上线后仍需在 PostgreSQL 核对表、索引、check constraint、回填数量和迁移状态。

## 升级步骤

1. 记录部署前 SHA、`VERSION`、镜像 digest、数据库规模、旧广场 public 数量和活动异步任务数量。
2. 备份 PostgreSQL、当前 OSS 配置和所有加密密钥；验证备份可读。
3. 在生产备份副本或 staging 数据库演练 `185`、`186`、`187`、`188`。
4. 先部署功能分支构建产物到单实例，观察迁移日志和 `image_library_migration_state`。
5. 确认旧广场已停止公开，旧内容接口不能绕过审核状态。
6. 在后台配置图片存储，执行真实 upload、HEAD/read 和 delete 连接测试。
7. 配置异步运行参数与图库保留/配额。
8. 确认图库维护 Worker 和持久异步 Worker 已运行，没有租约冲突或错误循环。
9. 先用测试用户、测试 Key 启用一个分组，完成工作台实时本机投稿→审核→同步、异步归档投稿审核、举报和清理闭环。
10. 核对真实 UsageLog、余额/订阅扣减、任务费用和 OSS 对象引用后再扩大范围。

不要在存储不可用时启用持久异步；不要在旧广场迁移尚未可解释时批量批准投稿。

## OSS 设置

全站使用一个当前 provider：

```text
qiniu / aliyun / tencent / custom_s3
```

后台接口：

```text
GET/PUT /api/v1/admin/backups/image-storage
POST    /api/v1/admin/backups/image-storage/test
```

测试连接必须真实执行小对象上传、HEAD/读取和删除；任一步失败都不能把配置保存为可用。生产密钥优先通过环境变量或密钥系统注入，不得提交到 Git。

切换存储身份前必须先处理历史对象。存在非 `deleted` 图片对象、任意 SC 输入对象，或带 object intent 的 upload reservation 时，服务端拒绝改变 provider、bucket、endpoint、region 或 path-style 寻址模式；应等待清理或迁移对象并原子更新引用。按历史存储身份解析旧凭证的完整 resolver 仍是 P1。

|---|---:|---|
| `public_base_url` | 空 | 提交响应中的外部查询根地址 |
| `worker_concurrency` | `4` | 每实例 Worker 数，硬上限 64，修改后需重启 |
| `worker_lease_seconds` | `120` | Redis/PG 租约阈值，最小 45 秒 |
| `recovery_interval_seconds` | `30` | 恢复扫描间隔 |
| `execution_timeout_seconds` | `900` | 单次上游调用超时 |
| `storage_retry_attempts` | `5` | OSS 后处理最大尝试 |
| `billing_retry_attempts` | `10` | 固定账单最大尝试 |
| `retry_backoff_seconds` | `30` | 后处理退避基数 |
| `download_max_bytes` | `33554432` | 单张远程参考图 32 MiB |
| `download_max_pixels` | `40000000` | 单张参考图 40 MP，硬上限 80 MP |
| `max_reference_images` | `8` | 单任务参考图数量，硬上限 16 |
| `max_reference_total_bytes` | `67108864` | 单任务参考图总字节 64 MiB，硬上限 256 MiB |
| `max_reference_total_pixels` | `80000000` | 单任务参考图总像素 80 MP，硬上限 320 MP |

| `download_max_redirects` | `3` | 每跳重新做 SSRF 检查 |
| `upload_timeout_seconds` | `300` | SC 写 OSS 总超时，最大规范化为 `600` 秒 |
| `upload_per_minute` | `20` | SC 每 API Key 每分钟上传尝试，最大规范化为 `1000` |
| `max_input_bytes_per_key` | `1073741824`（1 GiB） | 未过期输入、活跃 reservation 和未清理 intent，最大规范化为 100 GiB |

## 图库运行参数

| 配置 | 默认值 |
|---|---:|
| `retention_days` | `90` |
| `max_items_per_user` | `1000` |
| `max_bytes_per_user` | `5368709120`（5 GiB） |
| `max_image_bytes` | `20971520`（20 MiB） |
| `max_image_pixels` | `40000000`（40 MP） |
| `signed_url_expiry_seconds` | `3600` |
| `import_per_minute` | `20` |
| `publish_per_minute` | `10` |

这些字段和 `async_image` 一起保存在现有图片存储设置中。零值旧配置会规范化为安全默认值。后台设置优先于文件/环境 fallback；下载预算、超时和重试等动态策略保存后生效，但异步 Worker goroutine 数仍按进程启动值确定。

`download_max_bytes` 同时约束 SC 单图/请求有效图片负载，默认 32 MiB、硬上限 64 MiB；`download_max_pixels` 约束解码后的单图像素，三项 `max_reference_*` 约束整个生成任务。multipart wire body 只额外允许固定协议开销。`Idempotency-Key` 是客户端可选请求头，不是后台配置，最长 255 字节。

前端：

```bash

```

```bash
cd ../backend

go build -tags embed -ldflags="-X main.Version=${VERSION}" -o sub2api ./cmd/server
```

当前 Windows 主机可使用非仓库目录的 Go `1.26.5` 便携工具链；本机没有 Docker/可用 WSL 时，PostgreSQL/testcontainers 集成测试必须交给 Fork GitHub Actions，不能因此跳过。

`2026-07-22` 已在该便携工具链对合并上游后的代码完成 `go generate ./cmd/server`、强制 unit/default 全包测试和独立 server build；具体耗时与外部环境待办见 [测试与验收记录.md](测试与验收记录.md)。

## 多实例注意

- `worker_concurrency` 是每实例数量，总异步并发随实例数线性增长。
- 持久异步任务依靠 Redis 租约和 PostgreSQL CAS；图库维护依靠 PostgreSQL lease version 和心跳。
- 上线前用真实 PostgreSQL 验证多 Worker claim、stale running 恢复、租约丢失取消和 Outbox 重放。
- server cleanup 已挂接持久异步 Handler 和图库维护服务的 `Stop()`，并等待后台循环退出；真实多实例关闭时的数据库/Redis 行为仍需部署验收。

## 回滚

### 应用回滚前

1. 关闭所有分组的 `allow_async_image_generation`，阻止新持久异步任务。
2. 暂停用户导入/投稿入口或进入维护模式。
3. 等待 `queued/invoking/uploading/billing_pending` 清空；对 `storage_failed/billing_failed` 保存 staging 与固定账单。
4. 记录旧广场迁移游标、migrated/quarantined 计数、图库 Outbox、清理任务、`deleting` 对象，以及 SC upload attempts/reservations/object intents 数量。
5. 备份数据库和 OSS 清单。

### 回滚原则

- 不要直接删除 `185/186/187/188` 新表或回滚列；旧二进制忽略新表比丢任务、账单、输入所有权、幂等墓碑、延期投稿队列和对象引用安全。
- 不要恢复旧 `image_plaza_items` 的 public 标记，否则会重新暴露未经校验/审核内容。
- 不要删除 staging、固定账单、`image_storage_objects`、SC input objects、upload reservations 或 URL aliases，否则后续版本无法安全续跑、清理 intent 或判断所有权。
- 旧同步和旧 Redis 异步接口独立，可继续服务。
- 必须回退二进制时保留新表和 OSS 对象，待兼容版本恢复后继续任务、迁移和清理。

### 前滚恢复

重新部署兼容版本后：

1. 确认加密密钥和 OSS 配置没有变化。
2. 观察持久异步恢复扫描和图库 lease reclaim。
3. 核对 `execution_unknown` 没有被自动重调。
4. 核对 `library_archive`、Outbox、cleanup、SC stale intent 和旧迁移从持久状态继续。
5. 用抽样任务检查没有重复生成、重复扣费、重复输入/结果对象、重复图库或重复投稿，并确认旧上传幂等键只重签原对象。
