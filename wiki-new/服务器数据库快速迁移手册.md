# 服务器数据库快速迁移手册

本手册用于把正在运行的 Sub2API PostgreSQL 从旧服务器迁移到新服务器，并尽量缩短最终停写窗口。配套脚本：

```text
deploy/db-fast-cutover_ZJ.sh
deploy/sql/db_fast_cutover_verify_ZJ.sql
```

脚本和 SQL 带 `_ZJ`，表示本 Fork 自研运维工具，不属于原作者数据库迁移编号。

## 先说明不能做的事

普通 `pg_dump` 备份不包含可继续回放的 WAL/LSN。把一小时前的 dump 恢复到新服务器后，仅执行以下 SQL 不可能无损恢复最后一小时：

```sql
UPDATE users SET balance = ...;
UPDATE api_keys SET quota_used = ...;
```

原因是一次计费事务同时涉及 `users.balance/frozen_balance`、`api_keys.quota_used`、`user_subscriptions`、`user_platform_quotas`、`usage_billing_dedup`、`usage_logs`，还可能涉及支付订单、返利流水、批量生图和持久异步任务。只覆盖余额和 Key 会造成余额已经扣除但账务幂等键缺失，目标站重试后再次扣费；反过来也可能出现已有用量记录但余额没有扣除。

如果备份系统具备 PostgreSQL 物理备份和连续 WAL 归档，应优先使用 PITR 或流复制，那是大库近零停机的正确方案。本脚本面向只有 `pg_dump`、但源数据库在切换时仍可连接的通用场景。

## 快速方案原理

```text
T-1 小时：完整 dump -> 新服务器隔离恢复 -> 同版本迁移 -> 停止目标应用
切换时：停止源应用 -> 数据库级只读 -> 最终一致性 data-only 并行快照
       -> 清空目标业务数据 -> 并行恢复 -> 序列/账务摘要校验
       -> 同步配置、密钥、data、Redis -> 启动目标 -> 验收 -> 切流量
```

旧备份用于提前完成数据库创建、扩展、表结构和迁移检查。最终窗口只传输和恢复业务数据，不重新构建应用镜像或研究迁移错误。对超大数据库，最终 data-only 仍可能较慢，应升级为 WAL/PITR；不能用“只更新几个金额字段”换取表面速度。

## 前置条件

- 源库和目标库使用相同 PostgreSQL 大版本。
- 目标库已经恢复旧备份，并用与源站相同的 Sub2API 提交启动过一次以应用迁移，随后停止目标应用。
- 源站和目标站的 `schema_migrations`、public 列/约束/业务触发器、扩展、数据库编码与排序规则、应用版本一致。
- 操作机安装同大版本的 `psql`、`pg_dump`、`pg_restore`、`sha256sum` 和 `diff`。
- 数据库账号能读取源库全部 public 表，并能清空、恢复和 `ANALYZE` 目标库。目标迁移账号还必须能执行 `SET session_replication_role=replica`；官方 PostgreSQL 容器通常使用专用超级用户完成，托管数据库需先确认等效权限。
- `~/.pgpass` 权限为 `0600`，或使用 `pg_service.conf`；不要把密码写进命令行、脚本或 Git。
- 已备份并准备同步 `deploy/data/config.yaml`、`.installed`、JWT/TOTP/异步载荷加密密钥和 OSS 配置。
- 已决定 Redis 策略：继续连接原 Redis，或停写后复制 RDB/AOF。旧 Redis 24 小时异步任务不能只靠 PostgreSQL恢复。
- OSS 本身不在 PostgreSQL 中；切换 OSS 身份前必须满足对象引用保护规则。

生产操作前必须先在源库备份副本和隔离目标库完整演练。脚本包含 `TRUNCATE ... CASCADE`，目标确认值错误会直接破坏目标数据。

## 连接配置

推荐在操作机配置：

```ini
# ~/.pg_service.conf
[sub2api_source]
host=10.0.0.11
port=5432
dbname=sub2api
user=sub2api_migrator

[sub2api_target]
host=10.0.0.22
port=5432
dbname=sub2api
user=sub2api_migrator
```

密码写入 `~/.pgpass`：

```text
10.0.0.11:5432:sub2api:sub2api_migrator:源库密码
10.0.0.22:5432:sub2api:sub2api_migrator:目标库密码
```

```bash
chmod 600 ~/.pgpass ~/.pg_service.conf
export SOURCE_DSN='service=sub2api_source'
export TARGET_DSN='service=sub2api_target'
export JOBS=8
export WORK_ROOT=/srv/sub2api-migration
```

`JOBS` 通常设置为目标数据库 CPU 核数和可用 I/O 能力允许的值，建议从 `4` 或 `8` 开始。并发过高会放大磁盘和 WAL 压力，不一定更快。

## 第一阶段：提前恢复一小时前备份

在源服务器生成基础备份：

```bash
mkdir -p /srv/sub2api-base
pg_dump "$SOURCE_DSN" \
  --format=custom \
  --no-owner \
  --no-privileges \
  --file=/srv/sub2api-base/sub2api-base.dump
pg_restore --list /srv/sub2api-base/sub2api-base.dump >/dev/null
```

将 dump 传到操作机或目标服务器。先查看目标身份：

```bash
psql "$TARGET_DSN" -Atc \
  "SELECT current_database() || '@' || COALESCE(inet_server_addr()::text,'local') || ':' || COALESCE(inet_server_port()::text,'local')"
```

确认目标应用已停止，再执行：

```bash
export ZJ_CONFIRM_TARGET_ID='TARGET:sub2api@10.0.0.22:5432'
./deploy/db-fast-cutover_ZJ.sh restore-base /srv/sub2api-base/sub2api-base.dump
```

用准备上线的同一提交连接目标库启动一次，让程序自动执行尚未应用的迁移。目标站不能接公网流量。迁移完成后再次停止目标应用。

## 第二阶段：停机前预检

```bash
./deploy/db-fast-cutover_ZJ.sh preflight
```

脚本会拒绝以下情况：

- SOURCE 和 TARGET 实际指向同一个数据库。
- PostgreSQL 大版本不同。
- `schema_migrations` 文件名/checksum 指纹不同。
- public 表列、约束或业务触发器不同。
- 数据库编码、排序规则或扩展版本不同。

预检通过不代表可以切换，还必须核对磁盘空间。最终目录快照使用不压缩的并行格式，操作机可用空间建议至少为源数据库实际数据量的 `1.5` 倍。

## 第三阶段：最终切换

### 1. 停止源站写入

先从负载均衡摘除源站，然后优雅停止 Sub2API。Docker Compose 示例：

```bash
cd /opt/sub2api/source/deploy
docker compose -f docker-compose.local.yml -f docker-compose.override.yml stop sub2api
```

确认没有正在调用上游的异步生图任务。`invoking` 状态跨库迁移后可能转为 `execution_unknown`，脚本默认拒绝继续。

### 2. 数据库级冻结源库

先取得源库身份，再设置精确确认值：

```bash
export ZJ_CONFIRM_FREEZE_SOURCE='FREEZE:sub2api@10.0.0.11:5432'
./deploy/db-fast-cutover_ZJ.sh freeze-source
```

该命令设置数据库级 `default_transaction_read_only=on`，并终止其他客户端连接。即使旧应用或支付回调误重连，也不能继续写入。冻结后不要启动源应用。

### 3. 导出最终快照并覆盖目标数据

```bash
export ZJ_CONFIRM_REPLACE_TARGET='REPLACE:sub2api@10.0.0.22:5432'
./deploy/db-fast-cutover_ZJ.sh cutover
```

脚本依次执行：

1. 重新校验版本、迁移和列结构。
2. 确认源库只读、目标库没有其他客户端。
3. 使用 directory 格式和 `JOBS` 并行导出源库 public 全部业务数据。
4. 保存 manifest 和每个数据文件的 SHA-256。
5. 清空目标 public 根表并重置 identity/sequence。
6. 每个恢复连接使用会话级 `session_replication_role=replica`，避免并行写入时外键子表先于父表；连接结束后自动恢复，不永久禁用触发器。
7. 并行恢复最终数据快照并执行 `ANALYZE`。
8. 在源和目标分别运行 `db_fast_cutover_verify_ZJ.sql`。
9. 比较用户余额、冻结余额、Key 额度、订阅用量、平台额度、用量费用、账务幂等、支付、返利、异步任务、图片对象、设置指纹和序列安全线。

任何步骤失败都不会自动解除源库只读。目标覆盖开始前会写入 `restore.state`，只有校验完成才变成 `status=verified`。不要在 `status=incomplete` 的目标库上启动应用；修复原因后重新恢复基础备份并重做，或在目标尚未接受写入前回滚源站。

## 配置、文件和 Redis

数据库完成后、启动目标应用前同步：

```bash
rsync -aHAX --numeric-ids --delete \
  root@旧服务器:/opt/sub2api/source/deploy/data/ \
  /opt/sub2api/source/deploy/data/
```

必须人工核对：

- `config.yaml` 中目标 PostgreSQL/Redis 地址不能被旧配置覆盖。
- JWT、TOTP、异步请求载荷加密密钥必须保持原值，否则旧登录、2FA 或排队任务无法解密。
- 如果仍有旧本地 `image_plaza` 文件，必须随 `data/` 同步。
- 七牛、阿里、腾讯或 S3 凭证和 bucket 身份保持可解析历史对象。
- Redis 迁移需在源 Redis 停写后进行；持久异步任务可从 PostgreSQL恢复，但旧 Redis 临时任务、限流窗口和缓存不能假装已迁移。

## 启动目标和验收

```bash
cd /opt/sub2api/source/deploy
docker compose -f docker-compose.local.yml -f docker-compose.override.yml up -d postgres redis sub2api
docker compose -f docker-compose.local.yml -f docker-compose.override.yml logs -f --tail=200 sub2api
```

再次执行只读比较：

```bash
./deploy/db-fast-cutover_ZJ.sh verify
```

目标应用一旦开始产生新调用，源目标摘要自然会不同，此时不要再把差异当作迁移失败。切流前至少检查：

- 管理员和普通用户登录。
- 用户余额、冻结余额、累计充值。
- API Key 能鉴权，分组、额度和限流窗口正确。
- 订阅日/周/月用量、用户平台额度正确。
- 最近用量日志和账务幂等记录存在。
- 待支付/已支付/退款订单和返利流水正确。
- 异步任务、批量生图、OSS 图片和图库可查询。
- `schema_migrations` 包含当前全部 `_ZJ` 和上游迁移。

验收后再切换 DNS、反向代理或负载均衡。

## 回滚

目标还没有接受任何写入时，可以解除源库只读并重启旧站：

```bash
export ZJ_CONFIRM_UNFREEZE_SOURCE='UNFREEZE:sub2api@10.0.0.11:5432'
./deploy/db-fast-cutover_ZJ.sh unfreeze-source
```

目标一旦接受用户请求、充值回调或计费写入，就不能直接解除源库只读回滚，否则目标的新数据会丢失。此时必须反向执行同一套停写、最终快照和验证流程，或从明确的备份恢复。

## 性能建议

- 操作机尽量靠近两台数据库，避免公网高延迟。
- 使用同大版本 PostgreSQL 客户端。
- `WORK_ROOT` 放在本地 NVMe，不要放网络文件系统。
- `JOBS=4/8` 逐步压测，观察源库读取、目标库 WAL、磁盘队列和连接数。
- 最终快照使用 `--compress=0` 换取速度；迁移目录必须位于加密、权限受限的磁盘，完成后按备份策略销毁。
- 数据库很大或停机要求低于数分钟时，不要继续优化字段级 SQL，改用物理流复制或备份加连续 WAL/PITR。

## 边界

本工具没有在当前开发机真实执行双 PostgreSQL 迁移；当前环境缺少 Docker、可用 WSL 发行版和 Bash。上线前必须在隔离服务器演练，并记录源/目标 PostgreSQL 版本、数据量、耗时、摘要文件和回滚结果。
