# 发行版发布与安装操作手册

> 适用仓库：`JasonWangJie/sub2api`  
> 配套约定：`deploy/FORK_RELEASE.md`、`.cursor/rules/fork-release-deploy.mdc`

本文说明：**版本号在哪定义、怎么定版、怎么打 tag 出 Release 产物、产物出来后怎么一键安装/升级**。

---

## 1. 版本号在哪里定义

| 位置 | 作用 |
|---|---|
| `backend/cmd/server/VERSION` | **源码里的版本号**（无后缀数字，如 `0.1.162`）。编译进二进制，`sub2api --version` 会读它。 |
| Git tag（如 `v0.1.162`） | **发布开关**。推送 `v*` tag 会触发 GitHub Actions `Release` 工作流。 |
| GitHub Release 资产名 | 由 tag 推导：`sub2api_0.1.162_linux_amd64.tar.gz`（去掉 `v` 前缀）。 |
| `install.sh` / `upgrade.sh` | 通过 GitHub API 读 **本仓库** `releases/latest`，再下载对应 tar.gz。 |

关系简图：

```text
编辑 VERSION（可选） → git tag vX.Y.Z → push tag
        ↓
GitHub Actions Release（自动把 VERSION 写成与 tag 一致）
        ↓
上传 Release 产物（tar.gz + checksums.txt + 可选镜像）
        ↓
curl | bash 安装/升级
```

> 说明：Release 成功后，CI 会把 `backend/cmd/server/VERSION` 同步回 `main`，一般与 tag 去掉 `v` 后一致。日常发版以 **tag** 为准。

---

## 2. 后续版本号怎么定

规则（按优先级）：

1. **合并原作者仓库（`upstream` / `Wei-Shaw/sub2api`）后，若对方已有明确版本号**（对方 `VERSION` 或已发布 tag，如 `v0.1.164`），**本 Fork 跟对方版本号对齐**再发版。  
   - 例：上游到了 `0.1.164`，你合并后打 `v0.1.164`。
2. **上游没有可用版本号、或你只做了本 Fork 自有改动、尚未跟上游对齐时**，由你自行递增定义。  
   - 例：在 `0.1.162` 上仅改部署文档，可发 `v0.1.162`（首次）或自增补丁（如确需区分再约定）。
3. **tag 必须带 `v` 前缀**：`v0.1.162`（正确），`0.1.162`（不要用来触发本仓库 Release 约定）。
4. **不要重复推送已存在的同名 tag**。改已发版内容应使用新版本号。

当前基线：首次 Fork 发行版使用 **`0.1.162` / tag `v0.1.162`**。

## 3. 发版前检查

1. 代码已在 `main`，并已 `git push origin main`。  
2. GitHub 仓库已 **启用 Actions**（仓库页 → Actions → Enable）。  
3. 确认不会勾选 `simple_release`（简化发布**不上传**二进制，`install.sh` 会失败）。  
4. 可选：确认 `backend/cmd/server/VERSION` 与即将打的 tag 一致（不一致时 CI 仍会按 tag 覆盖并回写）。

```powershell
git checkout main
git pull origin main
Get-Content backend\cmd\server\VERSION
git status --short --branch
```

## 4. 打 tag + 推送（触发自动打包）

### 4.1 PowerShell（Windows 本机仓库）

把版本号换成你的目标版本（首次示例为 `0.1.162`）：

```powershell
$Ver = "0.1.162"          # 不带 v
$Tag = "v$Ver"            # 带 v

# 可选：发版前把 VERSION 写成一致
Set-Content -Path backend\cmd\server\VERSION -Value $Ver -NoNewline
git add backend/cmd/server/VERSION
git commit -m "chore: 同步 VERSION 为 $Ver"  # 若无变更可跳过
git push origin main

# 打 annotated tag 并推送（推送后自动跑 Release）
git tag -a $Tag -m "Sub2API $Ver 发行版"
git push origin $Tag
```

### 4.2 Bash（Linux / macOS）

```bash
VER=0.1.162
TAG=v$VER

echo -n "$VER" > backend/cmd/server/VERSION
git add backend/cmd/server/VERSION
git commit -m "chore: 同步 VERSION 为 ${VER}" || true
git push origin main

git tag -a "$TAG" -m "Sub2API ${VER} 发行版"
git push origin "$TAG"
```

### 4.3 盯构建

1. 打开：https://github.com/JasonWangJie/sub2api/actions  
2. 进入最新 **Release** 工作流，等待成功（绿勾）。  
3. 打开：https://github.com/JasonWangJie/sub2api/releases  

**install.sh 必需产物：**

- `sub2api_0.1.162_linux_amd64.tar.gz`
- `sub2api_0.1.162_linux_arm64.tar.gz`
- `checksums.txt`

若缺少上述文件：点进失败的 job 看日志；确认没有使用 simple_release。

### 4.4 仅手动重跑（tag 已存在时）

Actions → **Release** → **Run workflow**：

- `tag`：填已有 tag，如 `v0.1.162`
- `simple_release`：**不要勾选**

## 5. 产物出来后怎么装

目标机需要：Linux（amd64 或 arm64）、已安装并运行的 PostgreSQL / Redis、root（sudo）、`curl`、`tar`、Bash 4+。  
**不要**在用户机上 `git clone` + `go build` + `pnpm install`。

### 5.1 一键安装

```bash
curl -sSL https://raw.githubusercontent.com/JasonWangJie/sub2api/main/deploy/install.sh | sudo bash
```

脚本会：

1. 下载本仓库最新 Release 对应架构的 tar.gz  
2. 安装到 `/opt/sub2api/sub2api`  
3. 配置 systemd（`DATA_DIR=/etc/sub2api`）  
4. 启动服务；浏览器打开设置向导，生成 `/etc/sub2api/config.yaml`

### 5.2 一键升级

```bash
curl -sSL https://raw.githubusercontent.com/JasonWangJie/sub2api/main/deploy/upgrade.sh | sudo bash
```

升级会备份二进制与 `config.yaml`，替换程序并重启 systemd。

### 5.3 常用运维命令

```bash
sudo systemctl status sub2api
sudo journalctl -u sub2api -f
sudo systemctl restart sub2api
/opt/sub2api/sub2api --version
```

配置文件：`/etc/sub2api/config.yaml`

### 5.4 指定版本安装 / 回退

```bash
# 下载 install.sh 后执行，或：
curl -sSL https://raw.githubusercontent.com/JasonWangJie/sub2api/main/deploy/install.sh | sudo bash -s -- install -v v0.1.162
```

## 6. 合并原作者代码后的发版注意

1. 先合并 `upstream`，解决功能冲突。  
2. **恢复 Fork 身份**（仓库 URL、`upgrade.sh`、完整 Release 等），见 `deploy/FORK_RELEASE.md`。  
3. 读取上游 `backend/cmd/server/VERSION` 或上游最新 tag：  
   - 有 → 本 Fork 用同一版本号打 tag 发版；  
   - 无 → 自行定义下一版本。  
4. 再执行本文第 4 节打 tag + push。

---

## 7. 常见问题

| 现象 | 处理 |
|---|---|
| Actions 页要 Enable | 先启用 Actions，再推 tag |
| Release 成功但没有 `.tar.gz` | 是否误开 simple_release |
| `install.sh` 获取版本失败 | Release 未完成，或 API 限流（可设 `UPDATE_GITHUB_TOKEN`） |
| 版本号与二进制不一致 | 以 tag / Release 资产名为准；检查 CI 是否把 VERSION 同步回 main |
| 想删错 tag 重发 | 先确认无人依赖该版；`git push origin :refs/tags/vX.Y.Z` 后改内容再打**新** tag（谨慎） |
