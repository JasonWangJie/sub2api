## 日常发版（就这几步）

先确保改动已在 `main` 并已推送，GitHub Actions 已启用。然后：

```bash
# 确认已推到 main
git checkout main
git pull origin main

# 打 annotated tag（Release 说明会进 GitHub Release）
git tag -a v0.1.162 -m "Fork 发行版：一键安装/升级与完整 Release 产物"

# 推送 tag（这一步会触发 Actions 自动打包）
git push origin v0.1.162
```

下次发版只改版本号即可（见下方「版本号规则」）。

然后打开：

1. https://github.com/JasonWangJie/sub2api/actions → 等 **Release** 绿勾  
2. https://github.com/JasonWangJie/sub2api/releases → 确认有对应 `sub2api_*_linux_amd64.tar.gz` / `arm64` / `checksums.txt`

> CI 会按 tag 自动写入 `VERSION` 并上传产物，**不必**先手改 VERSION 再发版。

### 若页面像「只有源码 zip」——说明还没打出 Release

| 你看到的 | 含义 |
|---|---|
| Tags 页 + 只有源码 zip + 「Create release from tag」 | 只推了 tag，**Actions 没跑完或没跑** |
| Releases 页 + `github-actions` + 多个 `sub2api_*.tar.gz` | 打包成功 |

补跑：Actions → **Release** → **Run workflow** → 填 tag → **不要勾** simple_release。

## 版本号规则（支持四段）

`install.sh` / `upgrade.sh` 支持：

- 三段（跟上游）：`v0.1.162`
- 四段（本 Fork 中途更新）：`v0.1.162.1`

推荐节奏：

| 时机 | 示例 tag |
|---|---|
| 对齐原作者 `0.1.162` | `v0.1.162` |
| 自己又改了东西 | `v0.1.162.1`、`v0.1.162.2` … |
| 原作者到了 `0.1.163`，合并后再发 | `v0.1.163` |
| 合并后自己又改 | `v0.1.163.1` |

注意：

- tag 必须带 `v`；**不要**重复推已存在的同名 tag  
- 已发布的版本有问题才删 tag/Release 重推；正常更新请升号  
- 推新 tag 会自动触发 Actions，不必每次手动 Run workflow

## 产物出来后怎么装

```bash
# 安装
curl -sSL https://raw.githubusercontent.com/JasonWangJie/sub2api/main/deploy/install.sh | sudo bash

# 升级（会升到本仓库 releases/latest）
curl -sSL https://raw.githubusercontent.com/JasonWangJie/sub2api/main/deploy/upgrade.sh | sudo bash
```

目标机：Linux + PostgreSQL + Redis + sudo。

```bash
sudo systemctl status sub2api
/opt/sub2api/sub2api --version
# 配置：/opt/sub2api/config.yaml
```

---

## 合并原作者后

1. 合并 `upstream`，按 `deploy/FORK_RELEASE.md` 保住 Fork 身份  
2. 用上游版本号打三段 tag（如 `v0.1.163`）  
3. 之后自研再用四段（`v0.1.163.1`）

## 常见问题

| 现象 | 处理 |
|---|---|
| Actions 要 Enable | 先启用再推 tag |
| `startup_failure` | 多为 workflow `if` 写了 `secrets.*`（已修） |
| 没有 `.tar.gz` | 不要勾选 simple_release |
| install 拉不到版本 | Release 还没跑完 |
